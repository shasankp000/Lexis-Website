The Question

What does a language model actually learn?

Compression and prediction are the same problem. A model that predicts English well also compresses it well — that's just arithmetic coding. GPT-2 gets to about 1.30 bits per byte by learning statistical patterns from enormous amounts of text. But nobody has asked how much of that 1.30 is genuinely statistical discovery, and how much is just the model re-deriving things linguists already wrote down decades ago: that verbs inflect predictably, that pronouns corefer, that consonant clusters cluster.

Lexis-E answers this by building a compressor with zero learning, made entirely of explicit linguistic rules — and measuring how far it gets.

The Pipeline

Six cuts through one sentence

Lexis-E runs text through 12 stages. Below are six of them, applied to the same sentence in order. Each stage strips away a layer of predictable structure before handing a leaner representation to the next — like reading down through sediment, where each layer records something the one above it already assumed.

The Result

Beats classical compressors with zero statistics

Character-stream bits per byte on FineWeb, 100,000 characters. Lexis-E is slower than gzip by a wide margin and isn't meant to replace it — the point is that pure linguistic structure, with no corpus and no training, closes most of the distance to a trained neural model.

cmix and GPT-2 figures are published reference points measured on different corpora (enwik8 and WebText respectively) — included for orientation, not a controlled comparison.

Where the gains actually come from

The 5.25 bpb improvement over raw UTF-8 isn't one trick — it's five stages compounding on each other. Each stage reduces the entropy of what the next stage sees, so the gains multiply rather than add. Ordered roughly by where they sit in the pipeline:

01

Morphological decomposition

Storing (root, transformation code) instead of full inflected surface forms — "swimming" as (swim, GERUND) rather than six raw characters.

≈15.1% char reduction
02

Mixed-radix phonetic decomposition

Encoding characters as (phonetic class, position, morphological role) triples and taking deltas between them, so phonetically-adjacent characters cost almost nothing to encode.

2.42× smaller deltas
03

Discourse coreference substitution

Repeated entity mentions collapse to a placeholder token instead of being spelled out again. Costs a little on short documents, pays off past ~2,000 bytes.

net-positive above ~2KB
04

Online context mixing

Three linguistic evidence sources — character, morphological, and syntactic context — are blended by an adaptive model trained only on the document itself, and get sharper the longer the document runs.

improves with length
05

Case flag separation

Capitalization is stripped into a separate bitmap entirely, so the arithmetic coder only ever sees a flat, lowercase character distribution — one less source of variance to model.

simplifies char stream
Under The Hood

Four ideas that make it work

Lexis-E's compression ratio is a side effect of the real engineering problem: how do you turn linguistic knowledge into something an arithmetic coder can use? Four contributions carry most of the weight.

§4.1

Mixed-radix phonetic decomposition

A flat character ID throws away everything linguistics knows about that character — that 'b' and 'p' are both bilabial stops, that 's' at a word boundary usually marks a morpheme. Lexis-E instead encodes a (phonetic class, intra-class position, morphological role) triple per character, then only stores the delta between consecutive triples.

7 phonetic classes, deltas 2.42× smaller
§4.2

Factoriadic delta encoding

The delta stream from phonetic decomposition clusters tightly around zero. Rather than a fixed-width or Golomb code, Lexis-E encodes deltas in the factorial number system, where small integers get especially compact representations and larger ones grow slowly — matching the heavy-but-thin tail of the real distribution.

optimized for near-zero-heavy streams
§4.3

Anchor-based symbol slot splicing

Placeholder tokens for common words and coreferent entities need to be re-spliced back into the text on decode. Storing absolute offsets drifts on long documents; Lexis-E places anchors every 200 characters and interpolates between them instead.

zero drift past 10,000 characters
§4.4

Compact metadata envelope

The character stream is only part of the file — POS tags, morphological codes, and context tables all need storing too. A field-ID + length-prefix binary envelope, with the heaviest tables individually zstd-compressed, cuts that overhead roughly in half.

47–52% smaller full-payload size

Read plainly: if a billion-parameter model spends its training run rediscovering that "swimming" is "swim" plus a gerund marker, that's not new knowledge about the world — it's linguistics, re-derived at enormous computational cost. The residual 21.7% is presumably where the genuinely statistical, corpus-scale patterns live: idiom, register, world knowledge, the kind of thing no rulebook fully captures.

In Context

Where this sits

Compression and language modeling have been formally equivalent since Shannon — any good predictor is a good compressor, via arithmetic coding. What's less explored is compression built entirely from linguistic rules rather than statistics learned from data. Here's how Lexis-E compares to what's come before:

To our knowledge, no prior system combines morphological decomposition, syntactic tagging, coreference resolution, phonetic organization, and context mixing into one linguistically-motivated pipeline — most existing work in this space picks one of these signals, not all of them together.

Honestly

What this doesn't do (yet)

This is a research instrument, not a production tool, and the paper is upfront about where it falls short.

Full-payload size stays high

Even after halving metadata overhead, the complete file is still roughly 4× larger than the character stream alone — the context tables are expensive to store.

Not byte-exact

Punctuation normalization in Stage 1 means decoded output is semantically identical to the input but not always byte-for-byte reconstructable.

Slow by design

The full pipeline takes about 9 minutes to compress 100,000 characters — this trades speed for interpretability, not the other way around.

The 78% figure isn't fully controlled

Lexis-E was measured on FineWeb, GPT-2's figure comes from WebText. Directionally robust, but not an apples-to-apples run on identical data — yet.

The more interesting open question is what happens if a small neural model is conditioned on Lexis-E's linguistic representations instead of raw tokens — spending its entire parameter budget on the residual ~22% that's genuinely statistical, rather than re-learning morphology from scratch. That's the direction the paper points toward next.
